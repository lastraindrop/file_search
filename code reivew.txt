一、 深度代码审查报告 (Code Review Findings)
[Critical] 1. DataManager 单例模式存在并发初始化数据竞争

文件位置: file_cortex_core/config.py (DataManager.__new__ 与 __init__)

问题描述: 双重检查锁定（Double-Checked Locking）实现存在严重漏洞。__new__ 中的锁仅保护了实例的创建，但 __init__ 中的 self._initialized 检查是无锁的。如果线程 A 创建了实例但尚未完成 __init__，线程 B 获取实例后发现 _initialized 仍为 False，将导致 self.data 被二次重置，覆盖线程 A 的修改。

改进建议: 将初始化逻辑移入 __new__ 的锁内部，彻底废弃 __init__ 中的状态判定。

[Critical] 2. FastAPI 异步与同步阻塞的严重混用 (Event Loop Blocking)

文件位置: web_app.py (所有带有物理 I/O 的 async def 路由)

问题描述: 在 FastAPI 中，定义为 async def 的路由将强制在**主事件循环（Main Thread）**中运行。你在 /api/open, /api/fs/rename 等路由中直接调用了 os.scandir, os.rename, DataManager().save() 等同步磁盘 I/O 操作。在高并发下，这会导致整个 Web 服务器假死，WebSocket 断连。

改进建议: FastAPI 的最佳实践是：如果函数内有阻塞型同步 I/O，必须定义为常规 def（FastAPI 会自动将其放入外部线程池执行），或者使用 asyncio.to_thread。

[Critical] 3. 跨线程的 Asyncio Lock 滥用引发竞态条件

文件位置: web_app.py (websocket_action_stream 与 api_execute_tool)

问题描述: 全局变量 ACTIVE_PROCESSES 在 api_execute_tool 中使用 asyncio.Lock() (PROCESS_LOCK) 保护。但在 websocket_action_stream 内部的同步后台线程 run_stream 中，却直接执行了 ACTIVE_PROCESSES[proc.pid] = proc。同步线程无视了 asyncio 的锁，必然导致字典在迭代或并发写入时抛出 RuntimeError: dictionary changed size during iteration。

改进建议: 移除 asyncio.Lock，改用 Python 标准库的 threading.Lock()，以确保跨同步/异步边界的绝对安全。

[High] 4. 多线程搜索引发的线程池爆炸 (Thread Starvation)

文件位置: file_cortex_core/search.py (search_generator)

问题描述: 每次调用 search_generator 都会新建一个 ThreadPoolExecutor(max_workers=os.cpu_count() or 4)。如果前端有 10 个用户同时发起 WebSocket 搜索，系统会瞬间创建 10 * 16 = 160 个线程，导致严重的上下文切换开销和内存飙升。

改进建议: 在模块级别声明一个全局的 GLOBAL_SEARCH_EXECUTOR，所有搜索任务共享此线程池，限制系统总并发量。

[High] 5. Tkinter 主线程阻塞 (UI Freezing)

文件位置: file_search.py (update_stats)

问题描述: update_stats 内部调用了 FileUtils.flatten_paths，该方法包含递归 os.walk。如果 Staging List 中包含诸如 node_modules 这样巨大的目录，UI 线程将彻底卡死数秒。

改进建议: update_stats 必须放入后台线程执行，计算完毕后通过 self.root.after(0, update_label) 回调更新 UI。

[Medium] 6. 大文件字符编码推断导致内存溢出 (OOM Risk)

文件位置: file_cortex_core/utils.py (read_text_smart)

问题描述: from_path(file_path).best() 会将整个文件读入内存进行推断。如果遇到 2GB 的日志文件（即使用户尝试强制读取），会导致程序崩溃。

改进建议: 只读取前 32KB 进行编码探测，随后以流式或按行读取。

[Low] 7. 违反 PEP 8 规范 (Code Style)

文件位置: 全局多处

问题描述: 大量存在 if not path.exists(): return False 或 for i in range(5): do_something(); do_other() 这种将控制流写在同一行的做法。这严重降低了代码的可读性和调试时的断点准确性。

改进建议: 强制换行，保持每行一条语句。

二、 现阶段系统重构与验证计划 (Action Plan)

鉴于发现的底层并发和异步阻塞问题较多，现制定以下细致且全方位的修复与验证计划。必须严格按顺序执行，以防止系统发生回归崩溃。

阶段 1：并发隔离与状态机加固 (Core Synchronization)

工作目标: 解决 DataManager 单例竞态、FastAPI 同步阻塞及跨域锁污染问题。

修改内容与位置:

file_cortex_core/config.py:

重构 DataManager.__new__，引入 _init_data()，并删除 __init__。

在 MUTABLE_SETTINGS 校验处增加类型强校验，防止恶意嵌套覆盖。

web_app.py:

将 PROCESS_LOCK = asyncio.Lock() 替换为 PROCESS_LOCK = threading.Lock()。

对所有读写 ACTIVE_PROCESSES 的地方，无论是 async 路由还是 run_stream 后台线程，统一使用 with PROCESS_LOCK:。

剥离所有的 async def（除了 WebSocket 和纯内存计算），将 /api/open, /api/fs/* 统一修改为常规 def，交托 FastAPI 底层线程池。

file_cortex_core/search.py:

将 ThreadPoolExecutor 提取为模块级单例 SHARED_SEARCH_POOL = ThreadPoolExecutor(max_workers=os.cpu_count())。

单元检测 (Unit Testing):

新建 tests/test_concurrency_core.py:

用例 1: test_datamanager_strict_singleton: 启动 100 个线程同时调用 DataManager()，断言它们的 id() 绝对一致，且 load() 只被触发一次。
